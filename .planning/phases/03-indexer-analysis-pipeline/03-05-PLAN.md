<objective>
Resolve unresolved:: Call and TypeRef edges by matching called/referenced names to exported symbols in the graph, and expand extension resolution to handle .js-to-.ts mapping.

Purpose: The ts-extractor creates Call edges targeting `unresolved::functionName` and TypeRef edges targeting `unresolved::typeName`. Currently resolve_edges() skips these entirely (line 198-199 of resolve.rs), so they are silently dropped by add_edge() when no matching node exists. This is the root cause of the 9% edge ratio on OversizeConnect (106/1165). Resolving these edges will dramatically increase the edge count and reduce false-positive dead code from ~75% to <25%.

Output: Updated resolve.rs with name-based symbol resolution for unresolved edges, expanded extension resolution, and updated crawl.rs wiring.
</objective>

  <action>
  Add a new public function `resolve_unresolved_edges` to resolve.rs that resolves `unresolved::name` targets to actual graph nodes:

  ```rust
  pub fn resolve_unresolved_edges(
      edges: &mut Vec<SymbolEdge>,
      graph: &CodeGraph,
  )
  ```

  Implementation:
  1. Build a `HashMap<String, Vec<String>>` mapping symbol name -> list of symbol IDs for all exported symbols in the graph. Iterate `graph.graph.node_indices()`, filter by `is_exported == true`, insert `node.name -> node.id`.

  2. Build an import context map: `HashMap<String, HashSet<String>>` mapping source file path -> set of imported file paths. Iterate edges where `kind == EdgeKind::Import`, extract source file (from source_id via split_id or strip pseudo-node prefix) and target file (from target_id via split_id). This is used for disambiguation.

  3. For each edge where `target_id.starts_with("unresolved::")`:
     a. Extract the symbol name: `target_id.strip_prefix("unresolved::").unwrap()`
     b. Look up in the exported symbol HashMap
     c. If exactly one match: rewrite `edge.target_id` to the matched symbol ID
     d. If multiple matches: disambiguate using import context — check which candidate's file_path appears in the source file's import set. If one match, use it. If zero or multiple, use the first (sorted alphabetically for determinism).
     e. If zero matches: leave unchanged (will be dropped by add_edge, correct for external/third-party calls)

  Also expand the `resolve_extension` function's candidate list. After the existing `.ts`, `.tsx`, `/index.ts`, `/index.tsx` candidates, add JS-to-TS mapping candidates:
  - If the path ends with `.js`, also try replacing `.js` with `.ts`
  - If the path ends with `.jsx`, also try replacing `.jsx` with `.tsx`
  - If the path ends with `.mjs`, also try replacing `.mjs` with `.mts`
  - If the path ends with `.cjs`, also try replacing `.cjs` with `.cts`

  Modify the top of `resolve_extension` to handle this BEFORE the "already has known extension" early return: if the path ends in `.js`/`.jsx`/`.mjs`/`.cjs`, try the TS equivalent first, and only fall through to return the original if no TS version exists in the graph.

  Add `/index.js`->`.ts`, `/index.jsx`->`/index.tsx` to the extensionless candidate list as well.
