Replace the placeholder `crates/indexer/src/resolve.rs` with full implementation. The module has three responsibilities:

**1. TsConfigAliases — tsconfig.json path alias loading and resolution (PARS-10, D-28)**

```rust
use std::collections::HashMap;
use std::path::{Path, PathBuf};
use serde_json::Value;
```

Create `pub struct TsConfigAliases` with fields:
- `aliases: HashMap<String, Vec<String>>` — maps alias prefix (e.g., `"@/"`) to real path prefixes (e.g., `["src/"]`)
- `base_url: Option<String>` — tsconfig baseUrl if present

Implement:
- `TsConfigAliases::load(project_root: &Path) -> Self` — reads `project_root.join("tsconfig.json")`, parses JSON with `serde_json::from_str`. On file read error or JSON parse error, return `Self { aliases: HashMap::new(), base_url: None }` (D-13 graceful fallback; see RESEARCH.md Pitfall 1 for JSONC comment handling). Extract `compilerOptions.paths` as object, iterate key-value pairs: for each key like `"@/*"`, strip trailing `*` to get prefix `"@/"`, and for each value array entry strip trailing `*` to get resolved prefix. Store in `aliases` map.
- `TsConfigAliases::resolve(&self, raw_path: &str) -> String` — iterate `self.aliases`, if `raw_path.starts_with(prefix)`, substitute prefix with first resolved target and return. If no alias matches, return `raw_path.to_string()` unchanged.

**2. Path normalization — resolve raw import paths to canonical project-relative paths (Pitfall 5)**

Create `pub fn normalize_import_path(source_file: &Path, raw_import: &str, project_root: &Path) -> PathBuf`:
- If `raw_import` starts with `./` or `../`, it is a relative path: join with `source_file.parent()` and normalize `..` segments using `Path::components()` iteration (do NOT use `canonicalize` — it requires files to exist on disk, per RESEARCH.md A3 note). Iterate components: push normal/root components, pop on `ParentDir`, skip `CurDir`.
- If `raw_import` does not start with `.`, it is either an aliased path (already resolved by TsConfigAliases before this function is called) or a bare module (node_modules). For bare modules, return the raw path as-is (they will not match any project node and the edge will be silently dropped).
- After joining, make the path relative to `project_root` using `strip_prefix`. If stripping fails, return the joined path as-is.
- Try TypeScript module resolution: if the path has no file extension, try appending `.ts`, `.tsx`, and `/index.ts` in order. For each, check if a node exists in the graph with that file_path prefix. This resolution is done during the edge resolution pass (see below), not in this function — this function just normalizes the path.

Create `pub fn resolve_file_path(source_file: &Path, raw_import: &str, project_root: &Path, aliases: &TsConfigAliases) -> String`:
- First apply alias resolution: `let aliased = aliases.resolve(raw_import);`
- Then normalize: call `normalize_import_path(source_file, &aliased, project_root)`
- Convert result to string with forward slashes (replace `\\` with `/` for Windows compat)
- Return the resolved string

**3. Barrel chain resolution — resolve ReExport hops to true source (PARS-09, D-25, D-26)**

The resolution operates on a Vec of collected SymbolEdges (not yet inserted into the CodeGraph). The algorithm rewrites Import edges so their `target_id` points to the true defining node instead of a barrel intermediate.

Create `pub fn resolve_edges(edges: &mut Vec<cgraph_core::SymbolEdge>, graph: &mut CodeGraph, project_root: &Path, aliases: &TsConfigAliases)`:

This function performs three passes:

**Pass A: Resolve all raw paths in edges to canonical file paths.**
For each edge in `edges`:
- Extract the file path portion from `source_id` (everything before the first `::`) — this is the source file
- Extract the raw path portion from `target_id` (everything before the first `::`) — this is the raw import path
- If the raw path starts with `.` or matches a tsconfig alias prefix: call `resolve_file_path(Path::new(source_file), raw_path, project_root, aliases)` and replace the raw_path portion in `target_id` with the resolved path
- Handle extension resolution: if the resolved path does not end with `.ts`/`.tsx`/`.js`/`.jsx`, try appending `.ts`, `.tsx`, `/index.ts`, `/index.tsx` and check which resolved path has a matching node in the graph (using `graph.get_index(format!("{}::{}", resolved_path_with_ext, symbol_name))`)
- For `unresolved::` prefixed targets (Call and TypeRef edges): leave unchanged — these will be resolved by a future intra-file symbol matching pass or dropped

**Pass B: Build a ReExport lookup map and resolve barrel chains (D-25, D-26).**
- Collect all ReExport edges into a `HashMap<String, String>` mapping `source_id -> target_id` (e.g., `"index.ts::useToggle" -> "hooks.ts::useToggle"`). This is the hop map.
- For star ReExport edges (`source_id` ending in `::*`), expand: look at the target file, find all SymbolNodes with `is_exported == true` in that file, and for each create an entry in the hop map from `"barrel_file::symbol_name"` to `"target_file::symbol_name"`.
- Mark barrel files: for each file that has ReExport edges as source, call `graph.mark_barrel_file(file_path)`.
- Iterative chain following with cycle guard: for each Import edge in `edges`, check if `target_id` exists in the hop map. If yes, follow the hop. If the new target also exists in the hop map, follow again. Use a `HashSet<String>` visited set to detect cycles (Pitfall 2: circular barrel re-exports). Max 20 hops as a safety bound. Replace the edge's `target_id` with the final resolved target.

**Pass C: Remove ReExport edges from the edges vec.**
After resolution, filter out all edges where `edge.kind == EdgeKind::ReExport` — they have been folded into the Import edges. The final graph should contain no ReExport edges per the success criteria ("the graph contains no intermediate barrel-only edges").

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| tsconfig.json -> resolve.rs | User-controlled JSON file parsed for path aliases; could contain malicious paths |
| barrel edges -> resolution | Extractor output drives resolution; circular patterns could cause infinite loops |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Elevation of Privilege | resolve.rs (tsconfig alias) | mitigate | After alias substitution, verify resolved path does not escape project root using `strip_prefix(project_root)`. If it escapes, discard the edge and log a warning. |
| T-03-05 | Denial of Service | resolve.rs (barrel loop) | mitigate | Visited set (`HashSet<String>`) in barrel chain resolution breaks cycles. Additional 20-hop safety bound prevents runaway resolution on pathological inputs. |
| T-03-06 | Denial of Service | resolve.rs (star expansion) | accept | Star re-export expansion is bounded by the number of exported symbols per file. In practice, files have tens to low hundreds of exports. No additional bound needed. |
| T-03-07 | Information Disclosure | resolve.rs (tsconfig parse error) | mitigate | On JSON parse failure, fall back to empty aliases and continue (D-13). Do not expose file contents in error messages. `eprintln!("warn: could not parse tsconfig.json")` with no content detail. |
</threat_model>
